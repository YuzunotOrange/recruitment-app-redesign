# Recruitment App FastAPI Backend

FastAPI backend for the recruitment management app.

## Implemented scope

- User registration/login with JWT bearer authentication
- bcrypt password hashing
- Authenticated `GET /auth/me`
- Company CRUD scoped to the authenticated user
- Event CRUD scoped to the authenticated user
- Notification and reminder settings APIs
- Dashboard summary derived from the authenticated user's companies and events
- SQLite for local development
- PostgreSQL support for production
- Alembic migrations for database schema changes
- pytest coverage using temporary SQLite databases

## Environment

Copy the example file:

```powershell
cd backend
Copy-Item .env.example .env
```

Important values:

```text
ENVIRONMENT=development
DATABASE_URL=sqlite:///./app.db
FRONTEND_ORIGIN=http://localhost:3000
JWT_SECRET_KEY=
BACKEND_CORS_ORIGINS=http://localhost:3000,http://127.0.0.1:3000
```

Local development can omit `JWT_SECRET_KEY`. Non-local environments must set a long random `JWT_SECRET_KEY` and `FRONTEND_ORIGIN`.

## Database migrations

The application no longer creates or updates tables at FastAPI startup. Update the database explicitly with Alembic before running the server.

Run all pending migrations:

```powershell
cd backend
alembic upgrade head
```

Show the current database revision:

```powershell
cd backend
alembic current
```

Create a new migration after changing SQLAlchemy models:

```powershell
cd backend
alembic revision --autogenerate -m "describe schema change"
alembic upgrade head
```

Review autogenerated migrations before committing them. The Alembic environment reads `DATABASE_URL` from the same settings as the app, so the same commands work with local SQLite and production PostgreSQL URLs.

### Existing pre-Alembic databases

For a new empty database, run `alembic upgrade head`. If a database was already created by the previous `Base.metadata.create_all()` startup behavior, do not run the initial migration directly against it because the tables already exist. Use one of these paths instead:

- Local SQLite development data you can recreate: stop the server, back up or delete `app.db`, then run `alembic upgrade head`.
- Existing database you need to keep and whose schema already matches the models: back it up, then run `alembic stamp head` once to mark the current schema as the Alembic baseline.

After the baseline is stamped, future schema changes should use normal Alembic migrations and `alembic upgrade head`.

## Start

Start from an upgraded database:

```powershell
cd backend
alembic upgrade head
python -m uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

Open:

- Health check: `http://127.0.0.1:8000/health`
- Swagger UI: `http://127.0.0.1:8000/docs`

## Docker

Build the backend image:

```powershell
cd backend
docker build -t recruitment-backend:local .
```

Run with Docker Compose:

```powershell
cd backend
docker compose up --build
```

The container runs `alembic upgrade head` before starting uvicorn. Compose uses SQLite locally through the named volume `backend-sqlite-data`. PostgreSQL is not included in the local compose file.

Container environment variables:

- `ENVIRONMENT`: `development` locally, `production` on Render/AWS.
- `DATABASE_URL`: SQLite locally, PostgreSQL URL in production.
- `JWT_SECRET_KEY`: required for containers and production; use a long random value.
- `FRONTEND_ORIGIN`: frontend origin allowed by CORS.
- `PORT`: optional container port for uvicorn, defaults to `8000`.

## Production deployment notes

For production deployment, set production environment variables on the hosting service:

```text
ENVIRONMENT=production
DATABASE_URL=postgresql://user:password@host:5432/recruitment_app
FRONTEND_ORIGIN=https://your-frontend-domain.example.com
JWT_SECRET_KEY=replace-with-a-long-random-secret
```

Notes:

- `DATABASE_URL` controls the database backend. SQLite works locally; PostgreSQL is supported through `psycopg2-binary`.
- In non-local environments, CORS allows only `FRONTEND_ORIGIN`.
- `JWT_SECRET_KEY` is required outside local development.
- Run `alembic upgrade head` as part of deployment before serving traffic. The provided Docker image does this in its default command.

## Render deployment

The backend Dockerfile is compatible with Render Web Services. It starts by running migrations, then uvicorn on `0.0.0.0` and respects Render's `PORT` environment variable.

Recommended Render settings:

- Service type: Web Service
- Runtime: Docker
- Root Directory: `backend`
- Dockerfile Path: `backend/Dockerfile` if deploying from the repository root, or `Dockerfile` if the Render root directory is already `backend`
- Health Check Path: `/health`

Required Render environment variables:

```text
ENVIRONMENT=production
DATABASE_URL=postgresql://user:password@host:5432/database?sslmode=require
FRONTEND_ORIGIN=https://your-frontend-domain.example.com
JWT_SECRET_KEY=replace-with-a-long-random-secret
```

After deployment, verify `https://your-render-service.onrender.com/health` returns `{"status":"ok"}`.

## Supabase PostgreSQL

The backend can use Supabase PostgreSQL in production through `DATABASE_URL`. SQLite remains the default when `DATABASE_URL` is not set.

Example `.env` for Supabase:

```text
ENVIRONMENT=production
DATABASE_URL=postgresql://postgres.your-project-ref:your-password@aws-0-your-region.pooler.supabase.com:6543/postgres?sslmode=require
FRONTEND_ORIGIN=https://your-frontend-domain.example.com
JWT_SECRET_KEY=replace-with-a-long-random-secret
```

Apply migrations before starting FastAPI:

```powershell
cd backend
alembic upgrade head
python -m uvicorn app.main:app --host 0.0.0.0 --port 8000
```

## API smoke check

Register:

```powershell
Invoke-RestMethod -Method Post -Uri http://127.0.0.1:8000/auth/register -ContentType application/json -Body '{"email":"taro@example.com","password":"password123","name":"Taro","graduation_year":2027}'
```

Login:

```powershell
$login = Invoke-RestMethod -Method Post -Uri http://127.0.0.1:8000/auth/login -ContentType application/json -Body '{"email":"taro@example.com","password":"password123"}'
$token = $login.access_token
```

Current user:

```powershell
Invoke-RestMethod -Method Get -Uri http://127.0.0.1:8000/auth/me -Headers @{ Authorization = "Bearer $token" }
```
